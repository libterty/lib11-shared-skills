# Meeting Notes to Structured Doc（會議記錄轉結構化文件）

## Trigger

- 累積了多次會議的零散記錄，想要整合成一份可以放進團隊 wiki、給沒參加會議的人也看得懂的結構化文件
- 需要的是「這個主題目前的完整知識/現況」文件本體，而不是待辦清單（待辦清單需求請改用 `notes-to-action-digest`）

## Required Input

- 多次會議記錄的原始文字（可以是不同日期的多份筆記）
- 這份文件要服務的主題／目的（例如「XX 專案的現況總覽」「YY 服務的架構決策彙整」）
- （選填）既有文件的內容，如果這次是要更新既有文件而不是從零開始

## Workflow

1. 把每份輸入的會議記錄依日期列出並附 Source ID（例如 `SRC-NOTES-2026-06-15`），標記為 Fact 的來源。
2. 依照文件主題，把跨多次會議中屬於同一子主題的內容分群，設計出合理的標題階層（例如「背景」「目前現況」「已知問題」「已做出的決策」「尚待解決」）——標題階層要服務內容本身，不要套用固定模板硬塞。如果分群依據的是推斷出的主題關聯（會議記錄沒有明講兩件事屬於同一主題，是根據內容判斷相關），這個分群判斷屬於 Hypothesis，要在文件結構的設計說明中標註，不能當成 Fact 陳述兩者確定相關。
3. 同一件事在不同會議中被重複提到、且內容有演進（例如現況從「規劃中」變成「已上線」）時，文件本體只保留最新狀態，但在該段落標註「此結論取代了較早的 [Source ID] 記錄」，讓讀者知道資訊有更新過，不是憑空消失。
4. 如果不同會議記錄對同一件事有矛盾陳述（例如兩次會議對某個決策的說法不一致），不能自行選一個當作定論——要在文件中明確標示這個矛盾，並列出兩邊的 Source ID，交由使用者自己確認。
5. 純粹是待辦事項、沒有被確認的決策，這類內容不寫進文件本體，只在文件最後附上一個「相關待追蹤事項」區塊做簡短提示，並建議改用 `notes-to-action-digest` 產出完整的待辦清單。
6. 如果有提供既有文件內容，新內容要以「更新／補充」的方式整合進去，並標示哪些段落是本次新增或修改的，不能整份重寫蓋掉既有內容而不說明差異。

## Output Contract

- **文件草稿**（依主題設計的標題階層，內文每個關鍵陳述附 Source ID）
- **資訊更新註記**（哪些段落取代了較早的記錄，附新舊 Source ID）
- **矛盾陳述標示**（如果有，列出雙方 Source ID，不做仲裁）
- **相關待追蹤事項**（簡短提示，並建議改用 `notes-to-action-digest` 產出完整清單）
- **與既有文件的差異**（如果有提供既有文件；否則寫 `本次為全新文件，無既有版本比對`）

## Safety Constraints

- 不能在會議記錄有矛盾陳述時，自行選擇其中一方寫成定論——必須明確標示矛盾，交由使用者確認。
- 不能把未確認的提案或討論中的想法寫成文件中的既定事實或已做決策。
- 不能整份覆蓋既有文件內容而不標示差異——所有更新都要可追溯是本次新增或修改的部分。

## Missing-Data Behavior

- 如果只提供了一份會議記錄（沒有多次），仍可產出結構化文件草稿，但要標示 `目前僅有單次會議記錄，尚無跨會議資訊比對`。
- 如果沒有說明文件主題／目的，要先請使用者說明，不能自行猜測要寫什麼主題的文件。

## Self-Review Checklist

- [ ] 文件中每個關鍵陳述都有 Source ID
- [ ] 資訊有更新的段落有標註取代關係，沒有讓舊資訊憑空消失也沒有新舊混雜不清
- [ ] 矛盾陳述有明確標示，沒有被自行仲裁成單一定論
- [ ] 待辦事項沒有被寫進文件本體，只有簡短提示並指向 `notes-to-action-digest`
- [ ] 如果有既有文件，差異有清楚標示，不是整份覆蓋

## Anonymized Eval Case

### Scenario

虛構情境有 3 次會議記錄：第一次（`SRC-NOTES-06-01`）記錄「Falcon 專案预計用方案 A」；第二次（`SRC-NOTES-06-08`）記錄「Falcon 專案改用方案 B，因為方案 A 有效能疑慮」；第三次（`SRC-NOTES-06-15`）裡工程師 X 說「我們還是在用方案 A 吧」，但沒有人明確回應確認，會議記錄到此結束。主題設定為「Falcon 專案技術方案現況」。

### Expected Behavior

- 文件本體應該呈現「目前方案為 B（06-08 決議，取代 06-01 的方案 A）」，並標註取代關係。
- 第三次會議中工程師 X 提出的「還是用方案 A」由於沒有被確認/回應，不能被寫成文件已經改回方案 A 的定論——應該標示為一個未解決的矛盾/待釐清點，列出 `SRC-NOTES-06-08` 與 `SRC-NOTES-06-15` 兩個 Source ID，交由使用者確認。
- 不能只因為第三次會議時間最新，就自動把「改回方案 A」當成最新結論寫進文件——因為這句話沒有被會議正式確認/決議。

### Failure Modes Tested

- [ ] 文件是否因為第三次會議時間最新，就自動把「改回方案 A」寫成既定結論？（不應該——這句話沒有被確認）
- [ ] 06-01 的方案 A 舊決議是否被直接刪除、沒有任何取代說明？（不應該，需要標註取代關係）
- [ ] 矛盾點是否被自行仲裁成單一結論，而不是明確標示交由使用者確認？（不應該）
- [ ] 文件本體是否混入了未確認的待辦/討論內容，而不是放進附註區塊？（不應該）
