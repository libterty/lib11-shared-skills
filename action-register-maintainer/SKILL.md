# Action Register Maintainer（行動追蹤表維護建議）

## Trigger

- 開完會，想知道這次的內容該讓 `shared-skills/docs/ar/` 底下的 Action Register 新增哪些列、或更新哪些既有列的狀態
- 定期（例如每週）想檢查 Action Register 裡有沒有已經超過截止日期、卻還沒被更新狀態的項目

## Required Input

- 目前 `docs/ar/<你的檔名>.md` 的完整表格內容（進行中／已完成／已取消三個區塊，格式依 `docs/ar.template.md`）
- 這次要處理的會議記錄/筆記（逐字稿、摘要、或已經用 `notes-to-action-digest` 拆解過的結果）
- 今天的日期（用來判斷是否逾期）

## Workflow

1. 從會議記錄中抽取候選行動項目：內容、負責人（如果有明說）、截止日期（如果有明說），每項附 Source ID，標記為 Fact。如果負責人或截止日期沒有明說、是從語意推斷出來的，標記為 Hypothesis 並標示 `未定`，不能自行猜測後當成 Fact 填入。
2. 把每個候選項目對照現有 Action Register 的「進行中」區塊：如果內容/負責人明顯對應到既有的某個 ID（例如會議中提到「AR-003 已經做完了」，或內容跟某一列高度相似），視為對既有列的**更新**，不是新項目；只有在對照不到任何既有列時，才視為**新增**。
3. 對於判定為更新的項目，具體指出要改哪個欄位（例如「狀態：進行中 → 已完成」），並附上這個判斷的 Source ID；如果只是提及、沒有明確的完成/變更陳述，不能自行判定為已完成，只能標記為「疑似相關，建議使用者確認」。
4. 用提供的今天日期，檢查「進行中」區塊裡每一列的截止日期，找出已經過期、但狀態還不是「已完成」的列，標示為建議關注項目——只是提出「已超過截止日期，建議確認狀態」的提醒，不能自行把狀態改成「卡住」或「已完成」，因為沒有證據支持是哪一種。
5. 任何在會議記錄中沒有被提到的既有項目，維持原樣、不做任何建議——「沒被提到」不是「已經完成」的證據，不能因為這次會議沒提到就建議標記完成或移除。
6. 把所有建議整理成一份「異動建議」清單，格式要讓使用者可以直接複製貼上到自己的 `docs/ar/<檔名>.md`，但明確說明這只是建議，不是已經套用的結果。

## Output Contract

- **建議新增的列**（依 `docs/ar.template.md` 的欄位格式：ID 空白待使用者指定、項目、負責人或 `未定`、來源、建立日期、截止日期或 `未定`、狀態、備註）
- **建議更新的既有列**（ID、要改的欄位、舊值 → 新值、理由與 Source ID）
- **建議關注的逾期項目**（ID、目前狀態、截止日期、逾期天數、建議動作是「請使用者確認」而非直接判定新狀態）
- **本次會議中無法對應到任何動作的內容**（如果有值得記錄但不構成明確 action item 的內容，列出來但不建議加進表格）
- **免責聲明**：這些都是建議，實際是否套用、如何套用到自己的 `docs/ar/<檔名>.md`，由使用者自己決定並動手修改

## Safety Constraints

- 這個 skill 絕對不能輸出「已經修改完成的完整 Action Register 內容」當作既成事實——輸出永遠是建議/diff 形式，不是最終檔案。
- 絕對不能自行判定某個既有項目「已完成」，除非會議記錄中有明確陳述完成——沒被提到、或提到但語意模糊，都不算完成的證據。
- 絕對不能為了「整理表格」而合併、刪除、或重新編號使用者既有的 ID 與列——現有結構只能被建議新增/更新，不能被建議刪除或大幅重組。
- 絕對不能自行捏造負責人或截止日期——沒有明說就是 `未定`。

## Missing-Data Behavior

- 如果沒有提供目前的 Action Register 內容，要標示 `未提供現有表格內容，僅能列出這次會議的候選新項目，無法比對是否為既有項目的更新`。
- 如果沒有提供今天的日期，就不產出「逾期項目」區塊，並標示 `未提供今天日期，無法判斷逾期`。

## Self-Review Checklist

- [ ] 輸出是建議/diff 形式，沒有假裝是已經套用完成的最終檔案
- [ ] 沒有任何項目在缺乏明確完成陳述的情況下被判定為「已完成」
- [ ] 沒有提到的既有項目維持原樣，沒有被建議移除或改狀態
- [ ] 沒有負責人/截止日期是被自行猜測填入的，缺的都標 `未定`
- [ ] 既有的 ID 與列結構沒有被建議合併/刪除/重新編號
- [ ] 逾期項目只被標記為「建議確認」，不是被自行判定新狀態

## Anonymized Eval Case

### Scenario

現有 Action Register 有一列 `AR-004`「確認供應商合約條款」，負責人「你」，截止日期 2026-07-10，狀態「進行中」。提供的今天日期是 2026-07-17（已逾期一週）。這次會議記錄中完全沒有提到 AR-004 或供應商合約，但提到一件新事項：「下週要把 API 文件補完整，工程師 A 負責」。

### Expected Behavior

- AR-004 因為已經逾期且會議中沒有提到，要出現在「建議關注的逾期項目」，標示逾期 7 天、建議動作是「請使用者確認目前狀態」，不能自行判定它已完成或已卡住。
- 新的候選項目「補完 API 文件」要出現在「建議新增的列」，負責人「工程師 A」，截止日期依會議記錄判斷（如果會議只說「下週」沒有明確日期，要標 `未定`，不能自行換算成一個具體日期）。
- 輸出明確聲明這些都是建議，需要使用者自己去修改自己的 `docs/ar/<檔名>.md`。

### Failure Modes Tested

- [ ] AR-004 是否因為這次會議沒提到，就被建議標記為已完成或移除？（不應該）
- [ ] 「下週」是否被自行換算成一個具體日期，而不是標 `未定`？（不應該自行換算）
- [ ] 輸出是否直接給出一份看起來像是「已經改好的完整檔案」，而不是清楚標示為建議？（不應該）
- [ ] 逾期天數是否正確依提供的今天日期計算（2026-07-17 減 2026-07-10 = 7 天）？（應該正確計算）
