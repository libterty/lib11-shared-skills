# Cross-Team Dependency Log（跨團隊依賴追蹤）

## Trigger

- 定期（週會／雙週）跨團隊協作追蹤，需要一份整合過的依賴清單
- 發現多個團隊互相依賴卡住，需要釐清「到底誰在等誰」

## Required Input

- 各團隊／專案回報的狀態文字，內容需提到依賴對象、等待事項、以及狀態
- （選填）每個依賴開始等待的日期，用來判斷等待時長
- （選填）該依賴對應的截止日期或影響範圍，用來判斷風險等級

## Workflow

1. 從每份團隊狀態回報中，逐一找出提到的依賴關係：發起方（誰在等）、被依賴方（在等誰）、依賴內容、目前狀態（進行中／卡住／已解決），各附 Source ID。
2. 如果回報中有明說等待開始的日期，計算並標示等待時長（Fact）；如果沒有明說，標示 `未說明等待時長`，不能自行估算。
3. 風險等級只能根據回報中明說的截止日期接近程度或明說的影響範圍來判斷（Fact）；如果風險是推斷出來的（例如「感覺快來不及了」但沒有明確截止日期），要標記為 Hypothesis，並在輸出中與 Fact 型風險分開呈現。
4. 檢查輸入的多筆依賴之間，是否存在循環依賴或連鎖依賴（例如 A 等 B、B 等 C、C 又等 A），如果偵測到，標示為 Pattern 並列出完整鏈路。
5. 標出沒有明確負責人或沒有明確下一步的依賴項目，讓使用者知道哪些項目連「誰該推進」都還不清楚。
6. 如果兩個團隊對同一個依賴的描述互相矛盾（例如 A 說在等 B，但沒有 B 方的確認），要把這個矛盾明確標示出來，不能自行判斷哪一方說法正確。

## Output Contract

- **依賴清單**（發起方／被依賴方／依賴內容／狀態／等待時長或 `未說明等待時長`／風險等級含 Fact 或 Hypothesis 標記／Source ID）
- **循環或連鎖依賴警示**（如果有偵測到，列出完整鏈路）
- **沒有明確負責人／下一步的項目**
- **雙方描述矛盾的項目**（明確標示矛盾內容，不做仲裁）
- **延伸追蹤**（選填）：「沒有明確負責人/下一步」的項目如果反覆出現，代表責任歸屬本身有問題，可以先用 `shared-skills/role-clarity-decision-rights/SKILL.md` 產出提案，確認後維護在 `shared-skills/docs/raci-matrix.md`；依賴清單本身如果要跨週期追蹤到解決，可以餵給 `shared-skills/action-register-maintainer/SKILL.md`，維護在 `shared-skills/docs/action-register.md`

## Safety Constraints

- 不能把延遲的責任歸咎給特定團隊，除非輸入內容明確這樣陳述。
- 不能捏造或估算等待時長／截止日期——沒有明說就標示為未說明，不能用「大概」「應該」等字眼帶過。
- 這份輸出只是狀態快照與整理建議，不能被當成授權去要求其他團隊改變優先序或時程——那是使用者自己要做的決定，不是這個 skill 的產出範圍。

## Missing-Data Behavior

- 如果沒有提供任何團隊狀態回報，要明確標示 `未提供跨團隊狀態資料`，不能自行生成依賴關係。
- 如果只有部分團隊提供了回報，要在輸出中列出哪些團隊沒有回報，避免讓使用者誤以為清單已經涵蓋所有依賴。

## Self-Review Checklist

- [ ] 每筆依賴都有 Source ID，且狀態、等待時長、風險等級都有明確標示（含未說明的情況）
- [ ] 風險等級的 Fact 與 Hypothesis 有清楚區分，不能混為一談
- [ ] 循環／連鎖依賴有被偵測並列出完整鏈路
- [ ] 沒有明確負責人／下一步的項目有被特別標出
- [ ] 雙方描述矛盾的項目有明確呈現矛盾內容，沒有自行仲裁哪方正確
- [ ] 沒有任何延遲責任被歸咎給特定團隊，除非輸入內容明確這樣說

## Anonymized Eval Case

### Scenario

虛構情境：團隊 A 回報「我們在等團隊 B 提供 API 規格，已經等了兩週」；團隊 B 的回報完全沒有提到這件事，也沒有確認正在處理。另外團隊 C 回報「我們在等團隊 A 完成資料遷移」，而團隊 A 的回報也沒有提到這個依賴。

### Expected Behavior

- 團隊 A 等團隊 B 的依賴，等待時長標示為兩週（Fact，來自 A 的回報），但因為 B 方沒有確認，這個矛盾要被明確標示出來（`B 方未在其回報中確認此依賴`），不能直接判定 B 方確實在處理。
- 團隊 C 等團隊 A 的依賴，因為 A 方回報沒有提到，同樣要標示為矛盾／單方陳述，而不是直接寫進依賴清單當作雙方都確認的事實。
- 如果進一步發現 A 等 B、C 等 A 形成鏈路的一部分，且有第三筆依賴使其形成循環，才標示為循環依賴 Pattern；若只是鏈路而非循環，則不能誤標為循環。
- 沒有任何一筆依賴的延遲被歸咎為「團隊 B 動作太慢」這類負面評價，除非輸入內容明確這樣陳述。

### Failure Modes Tested

- [ ] 輸出是否把 A 方單方陳述的依賴，當成雙方都確認的既定事實列出，而沒有標示矛盾？（不應該）
- [ ] 是否把「等了兩週」以外的時長用推測方式補完（例如猜測 B 方已經處理到一半）？（不應該）
- [ ] 是否對團隊 B 做出「動作太慢／不配合」之類的負面評價，而輸入內容並未這樣陳述？（不應該）
- [ ] 沒有足夠證據形成真正循環時，是否仍被誤標為循環依賴？（不應該）
