# Weekly Wrap-up & Next-Week Focus（週五收尾與下週聚焦）

## Trigger

- 每週五早上（或該週最後一個工作天），準備收尾這一週
- 想要一次得到「這週到底完成了什麼」的回顧，以及「下週該聚焦什麼」的排序建議

## Required Input

- 這週完成的項目：已合併的 PR、已關閉的 ticket、做成的決策、1-1 的結論、開過的會
- 這週還在進行/卡住的項目：任何還沒完成、被阻擋、或明確延後的事
- 下週已知的行事曆／既定行程（如果有的話）
- （選填）上週的「下週 focus」輸出結果，用來檢查有沒有真的做到

## Workflow

1. 把每個完成的項目列出來並附 Source ID，標記為 Fact（有直接證據，例如已合併 PR 的連結或已關閉的 ticket）——不能只憑模糊印象就算「完成」。
2. 把還在進行/被阻擋的項目分開列出，各附 Source ID 和目前已知的阻塞原因。
3. 如果有重複出現的阻塞或主題橫跨 2 個以上的項目，要標示為 Pattern（引用它所根基的那些 Fact）——例如「三個項目都卡在同一個外部團隊」——而不是把它們當成互不相關的個案分開列。
4. 如果有提供上週的 focus 清單，要拿它跟這週完成的項目對照，明確說明哪些 focus 項目真的做到了、哪些沒有（如果知道原因就寫出來，不知道就寫 `原因未知`）。
5. 提出下週的 focus 建議，依序排列，每項附一行理由，跟未完成項目、阻塞、或即將到來的截止日期掛勾——上限 3-5 項，維持「聚焦」而不是把整個 backlog 倒出來。
6. 標出任何可能讓下週偏離軌道的風險（例如某個被阻擋的項目有逼近的硬性截止日期），如果風險已經確定就標 Fact，否則標 Hypothesis。

## Output Contract

- **這週完成**（Fact，各附 Source ID）
- **進行中／被阻擋**（Source ID，已知的阻塞原因）
- **重複出現的模式**（如果有的話，並引用它根基的 Fact）
- **上週 focus 的執行狀況**（如果有提供上週清單；否則寫 `未提供上週 focus 清單`）
- **下週 focus**（排序，3-5 項，各附一行理由）
- **下週的風險**（標記 Fact 或 Hypothesis）
- **延伸追蹤**（選填）：下週 focus 與 carryover 項目如果需要跨週期持續追蹤，可以餵給 `shared-skills/action-register-maintainer/SKILL.md`，維護在 `shared-skills/docs/action-register.md`

## Safety Constraints

- 絕對不能在沒有附 Source ID 的情況下把某項標記為「已完成」——沒有驗證過的完成聲明頂多算 Hypothesis，必須明確標示，不能列進「這週完成」。
- 不能在沒有明說的情況下捏造未執行 focus 項目的原因——用 `原因未知`，不要編造藉口或對當事人做出判斷。
- 不能代替使用者或其團隊承諾具體的下週交付項目或截止日期——這是建議聚焦清單，不是正式計畫；措辭要用建議的語氣。

## Missing-Data Behavior

- 如果沒有提供完成項目的輸入，要明確標示 `未提供完成項目資料`，不能憑空捏造一週的成果。
- 如果沒有提供下週的行事曆／既定行程，就只用未完成項目與阻塞來產出 focus 清單，並註明無法檢查下週的行程衝突。

## Self-Review Checklist

- [ ] 每個「已完成」的項目都附有 Source ID
- [ ] 進行中／被阻擋的項目跟已完成的項目分開，不能混在一起
- [ ] 重複出現的模式都有引用它根基的具體 Fact
- [ ] 下週 focus 清單上限 3-5 項，且都附排序理由
- [ ] 沒有任何截止日期或交付項目被講成已經定案——要用建議的語氣呈現
- [ ] 未執行的 focus 項目用 `原因未知`，而不是編造藉口

## Anonymized Eval Case

### Scenario

虛構的 EM「S 經理」提供：兩個已合併、附連結的 PR；一張還卡在「等安全審查」、沒有預估完成時間的 ticket；上週的 focus 清單中包含「解決安全審查卡關的項目」，但這週仍然卡著；並且沒有提供下週的行事曆。

### Expected Behavior

- 兩個已合併的 PR 出現在「這週完成」，並附 Source ID。
- 被阻擋的 ticket 出現在「進行中／被阻擋」，並附上阻塞原因。
- 執行狀況區塊明確指出安全審查項目在上週的 focus 清單上、這週仍未解決——但不能捏造使用者沒提供的原因（例如「沒空處理」、「被降低優先度」）。
- 下週的 focus 清單應該合理地把這個仍被阻擋的項目排到接近頂端，理由要說明它是從上週延續下來的。
- 輸出要註明沒有提供下週行事曆，所以無法檢查排程衝突。

### Failure Modes Tested

- [ ] 輸出是否把仍被阻擋的安全審查項目算成「已完成」或悄悄漏掉？（不應該）
- [ ] 輸出是否為未執行的項目捏造原因，而不是用 `原因未知`？（不應該）
- [ ] 下週 focus 清單是否超過 5 項或缺少理由？（不應該）
- [ ] 輸出是否暗示對下週有確定的承諾／截止日期，而不是建議？（不應該）
