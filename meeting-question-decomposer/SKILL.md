# Meeting Question Decomposer（會議核心問題拆解）

## Trigger

- 要開一場問題解決型會議，但只有一個模糊的核心問題（例如「我們是否應該在下季推出這個功能」），還沒拆成具體要問的子問題
- 過去的會議常常討論很熱烈，但事後發現真正該回答的核心問題沒有被回答（岔題）
- 想要在會議前確認：哪些問題今天真的要決定、哪些只是背景、哪些其實跟今天無關

## Required Input

- 這場會議要回答的核心問題／要做的決策（一句話描述）
- （選填）已知的限制條件、背景脈絡、之前討論過的內容
- （選填）會議總時長（用來判斷子問題數量是否合理）

## Workflow

1. 把使用者提供的核心問題原封不動陳述為 Fact，附 Source ID（例如 `SRC-USER-INPUT-核心問題`）——不能自行改寫成一個更大或更小的問題。
2. 把核心問題往下拆解成子問題，子問題之間盡量做到不重複、不遺漏（MECE）：每個子問題都要能回答「這個子問題如果有答案，會讓核心問題更接近被回答」，拆不出這個因果關係的子問題不要放進來。
3. 每個子問題再往下判斷：是「今天會議中需要當場討論決定」、還是「需要事先蒐集的資料/事實，不需要佔用會議時間現場討論」、還是「屬於執行細節或跟核心問題關聯薄弱，應該移到 parking lot 或另開會」。這個分類本身如果不是使用者明講的，屬於 Hypothesis（你的判斷），必須標示，讓使用者可以覆蓋。
4. 對於「今天需要當場討論決定」的子問題，依重要性與相依順序排列——需要先有答案的子問題排前面，避免討論到後面才發現前面的假設不成立。
5. 對於「需要事先蒐集資料」的子問題，列出具體要準備什麼資料、由誰準備（如果使用者提供了參與者角色的話；沒有提供就標 `未指定準備人`，不能自行指派）。
6. 對於「移到 parking lot」的子問題，簡短說明為什麼判斷它不屬於這場會議的核心範圍，讓使用者可以確認這個判斷，而不是悄悄捨棄。
7. 如果拆解出來「今天需要當場討論」的子問題數量明顯偏多（例如超過 5-6 個獨立子問題），要提醒使用者這可能代表核心問題本身還不夠聚焦，或者這場會議需要更長時間/拆成多場，而不是硬塞。

## Output Contract

- **核心問題**（原文重述，Fact，附 Source ID）
- **子問題拆解**（樹狀或分組列出，每個子問題附分類：今天決定／需要會前準備資料／parking lot，並標示 Fact 或 Hypothesis）
- **今天需要討論的子問題**（依討論順序排列，附排序理由）
- **會前需要準備的資料**（子問題、需要的資料、準備人或 `未指定準備人`）
- **Parking Lot**（子問題、移出理由）
- **聚焦度提醒**（如果「今天需要討論」的子問題數量明顯偏多，或跟提供的會議時長明顯不成比例，要提醒）
- **延伸使用**：把「今天需要討論的子問題」清單，可以直接當作 `shared-skills/meeting-agenda-draft/SKILL.md` 的會議目的輸入，產出附時間分配的議程

## Safety Constraints

- 不能把使用者沒提過的子問題，當成核心問題「應該」涵蓋的範圍自行加入——只能基於使用者提供的核心問題與背景脈絡做拆解與推論，且推論部分要標 Hypothesis。
- 不能把子問題分類（今天決定/會前準備/parking lot）當成確定的事實呈現——這是這個 skill 的判斷建議，使用者永遠可以覆蓋，必須讓使用者容易看出哪些是推薦、哪些不是。
- Parking Lot 的項目不能被悄悄捨棄不提——必須在輸出中明確列出並附理由，讓使用者自己確認是否同意移出。
- 不能自行指派會前資料的準備人，除非使用者有提供參與者角色資訊。

## Missing-Data Behavior

- 如果沒有提供核心問題，這個 skill 不能自行假設一個問題來拆解，要明確要求使用者先提供。
- 如果沒有提供會議時長，聚焦度提醒只能依子問題數量本身的合理性判斷（例如超過 6 個獨立子問題），並標示 `未提供會議時長，聚焦度提醒僅依子問題數量判斷`。

## Self-Review Checklist

- [ ] 核心問題有精確重述且附 Source ID，沒有被改寫
- [ ] 子問題拆解盡量做到不重複不遺漏，且每個子問題都能說明跟核心問題的因果關係
- [ ] 子問題分類（今天決定/會前準備/parking lot）中屬於推論的部分有標示 Hypothesis
- [ ] Parking Lot 項目都有列出且附移出理由，沒有被悄悄捨棄
- [ ] 沒有自行指派會前資料準備人，除非使用者提供了參與者角色
- [ ] 「今天需要討論」的子問題數量偏多時，有提出聚焦度提醒

## Anonymized Eval Case

### Scenario

虛構情境：使用者提供核心問題「我們是否應該在下季推出這個功能」，背景脈絡包含「上季客戶滿意度下降」「工程團隊目前有兩個專案在進行」，會議時長 45 分鐘。

### Expected Behavior

- 核心問題被精確重述，附 Source ID。
- 拆解出的子問題應該涵蓋類似「客戶是否真的需要這個功能」「對營收/留存的預期影響」「技術上是否可行、需要多少人力」「是否排擠現有兩個專案」等面向，且每個子問題都能說明跟核心問題的關聯。
- 「工程團隊目前有兩個專案在進行」這類背景事實，應該被辨識出跟「是否排擠現有專案」這個子問題有關，用來支持子問題的產生，而不是被忽略。
- 如果拆解出超過 6 個「今天需要討論」的子問題，且會議只有 45 分鐘，要明確提出聚焦度提醒，建議篩選最關鍵的幾個或考慮拆成多場會議。
- 純執行細節（例如「用哪個 UI 元件」）如果被提及，應該被歸類到 parking lot，並說明理由是「屬於執行層面，非本次核心決策範圍」。

### Failure Modes Tested

- [ ] 是否自行加入使用者完全沒提過的假設性子問題，而沒有標示為推論？（不應該，或至少要標 Hypothesis）
- [ ] Parking Lot 的項目是否被直接省略、沒有出現在輸出中？（不應該）
- [ ] 子問題分類是否被當成確定事實呈現，而不是可覆蓋的建議？（不應該）
- [ ] 子問題數量明顯偏多、時間明顯不夠時，是否仍然不提醒、直接硬排？（不應該）
