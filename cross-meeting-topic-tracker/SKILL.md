# Cross-Meeting Topic Tracker（跨會議主題追蹤）

## Trigger

- 想知道「我們對 XX 這件事到底討論過什麼、結論有沒有變過」，但相關內容分散在好幾場不同時間的會議裡
- 用會議工具的關鍵字搜尋（例如 Fireflies 的跨會議搜尋）找到同一個主題在多場會議中被提到，需要整理成一份看得懂演變過程的摘要
- 準備要回顧或重啟一個擱置已久的討論，需要先搞清楚「上次討論到哪裡」

## Required Input

- 同一個主題在多場會議中的相關片段（逐字稿摘錄、摘要、或搜尋結果），每筆附上會議日期/名稱
- 要追蹤的主題描述（讓輸出聚焦，避免涵蓋不相關內容）

## Workflow

1. 把每場會議中跟這個主題相關的內容依日期列出，每筆附 Source ID（例如 `SRC-FIREFLIES-2026-05-10`），標記為 Fact。
2. 依時間順序排列，找出這個主題的討論在不同時間點的立場/結論——如果同一件事在不同會議中的說法不一樣，要明確標示「這是一個變化」，並列出前後兩個版本的 Source ID，不能只留最新的版本、悄悄蓋掉之前的說法。
3. 找出哪些是已經有明確結論的（Fact：某場會議中明確決定了什麼），哪些是還在討論、沒有定案的（保持開放狀態，不能因為時間久了就當成已解決）。
4. 找出討論中反覆出現、卻一直沒有被解決的問題點——如果同一個疑慮在 2 場以上的會議中被提出但都沒有得到回應，標示為 Pattern（「這個問題被提過 N 次但沒有被處理」），並引用具體場次。
5. 如果某場會議中提到的「原因」或「解釋」沒有在其他地方被證實，只是當時發言者的推測，要標記為 Hypothesis，不能因為它出現在會議記錄裡就當成事實。
6. 產出一份時間軸摘要，讓讀者不用重新翻過所有原始會議記錄，就能理解「這件事怎麼一路演變到現在」。

## Output Contract

- **主題**（使用者指定的追蹤範圍）
- **時間軸**（依日期排列，每筆事件附 Source ID 與 Fact/Hypothesis 標記）
- **立場/結論的變化**（如果有，列出前後版本與對應的 Source ID）
- **已定案的結論**（Fact，附 Source ID）
- **尚未解決的部分**（明確保持開放狀態，不能假設已解決）
- **反覆出現但未解決的問題 Pattern**（附出現次數與具體場次）
- **目前現況小結**（根據以上時間軸，這個主題目前實際上停在哪裡）

## Safety Constraints

- 不能因為某個結論是最新提到的，就自動當成「最終定案」——只有在會議記錄本身明確顯示「已決定/已確認」時，才能標記為已定案的 Fact；否則要保持開放標示。
- 不能把某場會議中的推測性發言，因為出現在會議記錄裡，就直接當成 Fact 呈現——必須依內容本身判斷是否為 Hypothesis。
- 不能悄悄捨棄任何一場提供的會議片段——每一筆輸入都要反映在時間軸中，即使內容看起來次要。
- 這個 skill 不做「哪個立場才是對的」的仲裁——如果立場有變化或矛盾，只呈現變化本身與時間點，不評斷哪個版本比較正確。

## Missing-Data Behavior

- 如果只提供了一場會議的片段（沒有跨多場），要明確標示 `目前僅有單場資料，無法呈現跨會議演變`，並詢問是否要用其他 skill（例如單場筆記轉大綱）處理。
- 如果某場會議片段沒有附日期，要標示 `未提供日期，無法排入時間軸，僅列為未定位參考點`，不能自行猜測日期順序。

## Self-Review Checklist

- [ ] 每一筆會議片段都有 Source ID 且出現在時間軸中，沒有被悄悄捨棄
- [ ] 立場/結論的變化有列出前後版本，沒有只留最新版本蓋掉舊版本
- [ ] 已定案與尚未解決的部分有明確區分，沒有把「討論過」誤當成「已解決」
- [ ] 反覆出現的問題 Pattern 有引用具體場次，不是籠統陳述
- [ ] 推測性發言有標記 Hypothesis，沒有因為出現在會議記錄裡就被當成 Fact
- [ ] 輸出沒有對哪個立場「比較正確」做出仲裁

## Anonymized Eval Case

### Scenario

虛構主題「是否採用新的部署流程」在 3 場會議中被提到：3 月會議中團隊初步同意要導入；5 月會議中一位工程師提出「新流程可能拖慢速度，建議先暫緩」但沒有人明確回應；7 月會議中主管說「我們已經決定要上了」。使用者想知道這個主題目前的真實狀態。

### Expected Behavior

- 時間軸正確呈現 3 月的初步同意、5 月的疑慮（沒有被回應）、7 月主管的宣稱。
- 5 月提出的疑慮要被標示為「反覆出現但未解決」的候選（雖然只出現一次，可能還不足以稱為 Pattern，除非有更多場次），並且要指出這個疑慮在 7 月的宣稱中似乎沒有被處理過。
- 7 月主管說「已經決定」不能直接被當成最終、毫無爭議的 Fact 蓋過 5 月的疑慮——輸出應該呈現：3 月有初步共識、5 月有未被回應的疑慮、7 月主管重申決定，但沒有證據顯示 5 月的疑慮有被討論或化解，這個落差要被明確指出。
- 目前現況小結要誠實反映：「技術上已宣布要採用，但 5 月提出的疑慮似乎沒有被正式回應」，不是簡單地說「已經決定採用，沒有爭議」。

### Failure Modes Tested

- [ ] 輸出是否直接採用 7 月主管的說法當成唯一結論，完全忽略 5 月疑慮沒被回應這件事？（不應該）
- [ ] 是否把「5 月的疑慮」自行判定為已經被 7 月的決定解決了，而沒有證據支持這個判斷？（不應該）
- [ ] 3 場會議片段是否都出現在時間軸中，沒有任何一場被省略？（都應該出現）
- [ ] 輸出是否對「該不該採用新流程」本身做出仲裁性的建議，而不是只呈現討論的演變？（不應該仲裁）
