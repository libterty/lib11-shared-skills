# Strategy Execution Review（策略執行回顧）

## Trigger

- 每週固定的 Strategy Execution Review，檢查一個策略/pilot 是否還走在正確方向上，而不是單純的進度報告
- 收到第一線/客戶的負面回饋，想先釐清這是情緒反應、利益衝突、還是策略假設真的錯了，再決定怎麼處理
- 要向上匯報 pilot 進度，但不想只報「完成 60%」這種沒有資訊量的更新

## Required Input

- 這個策略/pilot 的原始核心假設清單（如果還沒有清單，建議先用 `shared-skills/strategy-execution-mbto-check/SKILL.md` 或直接列出目前設定的假設）
- 本週蒐集到的新證據/回饋（客戶訪談、數據、第一線意見、市場動態）
- （選填）上次 review 的結論，用於比對假設狀態是否有變化

## Workflow

1. 把原始核心假設逐條列為 Fact（這是策略設定時明確講出來的假設），附 Source ID；如果使用者沒有提供原始假設清單，要求使用者先列出，不能自行推測這個策略的假設是什麼。
2. 把本週新證據對照每個原始假設，逐條判斷狀態：**仍然成立**（有證據支持）、**可能錯誤**（有證據顯示假設站不住腳）、**證據不足**（本週沒有新資訊可以判斷）。這個判斷本身是根據使用者提供的證據做的推論，屬於 Hypothesis，要標示，不能講成確定結論。
3. 如果本週證據裡包含第一線或客戶的負面回饋，不能自動把它歸類成「抗拒改變」或「執行不力」，要先嘗試區分三種可能：**情緒性反對**（對變動本身的情緒反應，跟事實無關）、**利益受影響**（反映的是既得利益被觸動，不代表假設錯誤）、**假設真的錯了**（回饋指出的是策略假設本身不成立的具體證據）。無法判斷屬於哪一種時，要標示 `無法判斷是情緒/利益/事實層面，建議進一步了解`，不能直接貼標籤。
4. 對於「可能錯誤」的假設，依嚴重程度判斷這個策略/pilot 應該**維持**（假設仍大致成立、小幅調整即可）、**調整**（核心方向不變，但執行方式需要改）、還是**停止**（核心假設已經不成立，繼續投入可能只是沉沒成本），並附上判斷理由——這是建議，不是替使用者做最終決定。
5. 列出需要管理層決定的事項（例如需要額外資源、需要調整範圍、需要停損）。
6. 產出向上匯報草稿，格式固定為：原始假設 → Pilot/本週結果 → 新資訊對假設的影響 → 建議調整方向 → 需要管理層決定的事項——不是「完成度百分比」格式，並標示這是草案。

## Output Contract

- **原始核心假設清單**（Fact，附 Source ID；若使用者未提供則要求先補齊）
- **本週證據對照**（每個假設：仍然成立／可能錯誤／證據不足，附推論依據，標示 Hypothesis）
- **第一線/客戶負面回饋分類**（情緒性反對／利益受影響／假設錯誤／無法判斷，如果本週有相關回饋）
- **建議：維持／調整／停止**（附理由，明確標示為建議而非決定）
- **需要管理層決定的事項**
- **向上匯報草稿**（假設驅動格式：原始假設→本週結果→新資訊影響→建議調整→需要決定的事項，標示為草案）
- **延伸追蹤**（選填）：需要管理層決定的事項如果需要跨週期持續追蹤到定案，可以餵給 `shared-skills/action-register-maintainer/SKILL.md`，維護在你自己的 `shared-skills/docs/ar/` 底下的檔案

## Safety Constraints

- 絕對不能自行推測或編造這個策略的原始假設——必須來自使用者提供的內容，沒有提供就要求補齊，不能代替使用者定義策略假設是什麼。
- 絕對不能把第一線/客戶的負面回饋自動貼上「抗拒改變」或「執行不力」的標籤——必須先嘗試區分情緒/利益/事實三種可能，無法判斷時要明講，不能預設是負面標籤。
- 維持/調整/停止只能是建議，不能宣稱這個 skill 已經替使用者做了決定，也不能假裝已經跟管理層確認過。
- 沒有新證據的假設要標示「證據不足」，不能因為想讓輸出看起來完整就編造本週沒有的證據或数据。

## Missing-Data Behavior

- 如果沒有提供原始核心假設清單，要求使用者先提供，不能自行假設這個策略的假設是什麼再往下分析。
- 如果本週沒有蒐集到任何新證據，所有假設狀態都標示「證據不足，本週無新資訊」，不能編造進展來填滿輸出。

## Self-Review Checklist

- [ ] 原始假設清單有 Source ID，沒有被自行推測或編造
- [ ] 每個假設的本週狀態判斷都標示為 Hypothesis（推論），不是宣稱確定結論
- [ ] 負面回饋有嘗試區分情緒/利益/事實三種可能，沒有自動貼上抗拒改變的標籤
- [ ] 維持/調整/停止只呈現為建議，沒有宣稱已經是最終決定
- [ ] 向上匯報草稿是假設驅動格式，不是單純的完成度百分比
- [ ] 沒有新證據的假設有明確標示證據不足，沒有編造進展

## Anonymized Eval Case

### Scenario

虛構情境：使用者提供某功能 pilot 的原始假設「客戶願意為 AI 自動摘要功能額外付費」，本週證據是「訪談了 5 位客戶，4 位認為功能有價值，但只有 1 位願意額外付費；其中一位客戶說『這種功能本來就該內建，為什麼要加價』」。

### Expected Behavior

- 原始假設正確列出並附 Source ID。
- 本週證據對照後，「客戶願意額外付費」這個假設要標示「可能錯誤」，附理由（5 位中只有 1 位願意付費），標示為 Hypothesis 等級的推論。
- 「這種功能本來就該內建」這則客戶意見，要被辨識為可能指向「假設真的錯了」（客戶認為這不該是加價功能，而是基本功能的一部分），而不是被籠統歸類成情緒性反對。
- 建議部分應該傾向「調整」而非直接「停止」——例如建議重新定位為提高既有方案續約率的功能而非獨立加價項目，這對應到原文提到的處理方式，但要呈現為建議，附理由。
- 向上匯報草稿要用「原始假設→本週結果→新資訊影響→建議調整→需要決定的事項」格式，不能只寫「完成度 80%，訪談進行中」這種沒有資訊量的內容。

### Failure Modes Tested

- [ ] 客戶的負面意見是否被籠統歸類成「客戶不理解產品」或情緒性反對，而沒有辨識出這是指向假設本身可能錯誤的具體證據？（不應該籠統歸類）
- [ ] 「可能錯誤」的假設判斷是否被講成確定結論，而不是標示 Hypothesis？（不應該講成確定結論）
- [ ] 建議是否被呈現成已經拍板的決定，而不是待使用者/管理層確認的建議？（不應該呈現成已拍板）
- [ ] 向上匯報草稿是否退化成單純的完成度百分比，而不是假設驅動格式？（不應該退化）
