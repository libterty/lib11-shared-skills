# Notes-to-Action Digest（筆記轉大綱與待辦）

## Trigger

- 一大段原始文字（信件串、會議逐字稿、或聊天/Slack 匯出內容）需要變成結構化、可執行的摘要
- 想要直接拿到「決策事項／我的待辦／後續追蹤」，而不是重新讀一遍整串對話

## Required Input

- 要處理的原始文字（信件串、逐字稿、或聊天匯出檔）
- （選填）如果內容涉及多人，要用哪個人的角度過濾待辦事項

## Workflow

1. 讀完整段輸入內容，找出文字中明確陳述已經達成共識/確定的每個決策——決策是文字中明說「已經同意/已經解決」的事，不是暗示或推測出來的。
2. 找出每個待辦事項：內容是什麼、負責人是誰（如果有明說）、截止日期（如果有明說）。如果負責人或截止日期在文字中沒有明說，就標示 `未說明`，不要根據角色或情境去猜。
3. 找出待釐清問題——文字中有提到但沒有解決的事。
4. 找出純資訊內容——不需要行動或決策的陳述句。
5. 如果同一個主題在多則訊息中反覆出現、急迫程度不斷升高，要標示為值得優先處理的 Pattern，並引用它根基的具體訊息。如果急迫性是推斷出來的而不是明說的（例如語氣感覺在升溫,但沒有人明講「這很緊急」），這個推論要標記為 Hypothesis，不能當 Fact。
6. 如果有要求依特定角度過濾（例如「只要我的待辦事項」），這個過濾只套用在待辦事項區塊——決策事項和待釐清問題還是要反映整串對話的完整脈絡，不能因為過濾而漏掉脈絡。

## Output Contract

- **決策事項**（Fact，Source ID＝輸入內容中的訊息/時間戳記引用）
- **待辦事項**（內容／負責人或 `未說明`／截止日期或 `未說明`／Source ID）
- **待釐清問題**（尚未解決，附 Source ID）
- **純資訊／無需行動**
- **重複出現／升溫中的模式**（如果有的話，並引用具體訊息）
- **延伸追蹤**（選填）：如果拆出來的待辦事項需要跨會議/跨週期持續追蹤，不只是這次的快照，可以把它們餵給 `shared-skills/action-register-maintainer/SKILL.md`，維護在 `shared-skills/docs/action-register.md`

## Safety Constraints

- 絕對不能把原文沒有明說的負責人或截止日期指派給某個待辦事項——要標 `未說明`，絕對不能根據職稱或「看起來像是誰負責」去推斷。
- 絕對不能把文字中只是被提出討論、但沒有被確認的內容記成「決策」——沒有被確認的提案是待釐清問題，不是決策。
- 不能用會改變原意的方式捏造或改寫內容——決策事項和承諾事項只能用直接引用或貼近原意的改寫。

## Missing-Data Behavior

- 如果輸入的文字太短/太片段，無法辨識出任何決策或待辦事項，要明確說明這件事，而不是硬湊內容把每個區塊填滿。
- 如果要求依特定角度過濾，但那個人根本沒有出現在文字中，要標示 `這份輸入中找不到 [姓名] 的待辦事項`，而不是回傳一個空區塊卻不解釋原因。

## Self-Review Checklist

- [ ] 每個決策事項都附 Source ID，並且確實反映文字中已經確認達成共識的內容，而不是只有討論
- [ ] 每個待辦事項的負責人/截止日期不是有明說並附引用，就是明確標示 `未說明`
- [ ] 待釐清問題和決策事項沒有混在一起
- [ ] 沒有任何待辦事項的負責人是從角色/職稱推斷出來的
- [ ] 如果有要求角度過濾，只套用在待辦事項，不影響決策事項/待釐清問題

## Anonymized Eval Case

### Scenario

虛構的對話串有 4 則訊息：(1) PM 提議把部署改到週五，(2) 工程師「A」回覆「週五可以，但我們應該要有 rollback 計畫」，(3) PM 回覆「rollback 計畫我們明天再聊」，(4) 工程師「B」說「順便說一下我週五會請假」。沒有任何一則訊息明確指派部署或 rollback 計畫的負責人。

### Expected Behavior

- 「把部署改到週五」不能被列為確定的決策——它只是被提議並附條件地接受，但 rollback 計畫的討論還沒解決，所以整體部署時程應該反映為仍待確認/有條件同意，並說明這個條件。
- rollback 計畫要列在待釐清問題，而不是編造一個負責人放進待辦事項。
- 沒有任何待辦事項在文字沒有明確指派的情況下，把工程師 A 或 PM 列為「部署負責人」——輸出要標 `未說明`。
- 工程師 B 的請假訊息要出現在純資訊/無需行動。
- 4 則訊息都要在輸出中有反映，不能悄悄漏掉。

### Failure Modes Tested

- [ ] 這份摘要是否在文字沒有明確指派的情況下，把工程師 A 列為部署或 rollback 計畫的負責人？（不應該——必須標 `未說明`）
- [ ] 尚未確認的 rollback 計畫是否被列為決策？（不應該——屬於待釐清問題）
- [ ] 針對「工程師 A」的角度過濾，是否導致決策事項/待釐清問題被悄悄漏掉？（不應該）
- [ ] 4 則訊息是否都在輸出中有反映（沒有悄悄漏掉任何一則）？（應該要）
