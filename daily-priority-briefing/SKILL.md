# Daily Priority Briefing（每日優先序盤點）

## Trigger

- 每天開始工作前，還沒開始開會/回信之前
- 想要一份整合過的「今天到底該做什麼」清單，而不是分別去看行事曆、信箱、任務追蹤系統

## Required Input

- 今天的行事曆事件（時間、標題，需要的話含與會人員）
- 未處理的待辦項目：未讀/已標記的信件、被指派的 ticket、昨天沒做完的事——每項都要有足夠細節可以判斷急迫性
- 這些項目中明確提到的截止日期
- （選填）昨天的 carryover 清單，如果昨天也跑過這個 skill

## Workflow

1. 把每個輸入項目列出來並附上 Source ID（例如 `SRC-GCAL-standup-0930`、`SRC-GMAIL-thread-4821`、`SRC-LINEAR-ENG-123`）。
2. 把每個項目分類為 **Fact（事實）**（有明確的截止日期／負責人／明確請求）或 **Hypothesis（推測）**（推斷出來的急迫性，例如「寄件人是主管所以感覺很重要」但沒有明確截止日期）。
3. 依明確陳述過的優先序邏輯排序——結合明確截止日期的接近程度、對他人造成的阻塞影響、以及會議的固定時間限制。要說明排序依據，不能只丟出一份排好序但沒理由的清單。
4. 檢查行事曆有沒有重疊、背靠背沒有緩衝時間的會議、或是會議時間跟清單中較前面項目的截止日期衝突——這些都要明確標出來。
5. 在最終輸出中，把固定時間的項目（會議）和彈性時間的項目（任務）分開，讓使用者看得出哪些是可以調動的。
6. 任何無法自信分類的項目（沒有截止日期、沒有明確負責人、請求內容模糊）都要放進明確的「待釐清」區塊，而不是硬排進優先序清單。

## Output Contract

- **固定時間行程**（會議，依時間排序，各附 Source ID）
- **優先序待辦清單**（依序排列，每項附 Source ID、一行排序理由、Fact/Hypothesis 標記）
- **行事曆衝突／風險**（重疊會議、無緩衝的背靠背會議、截止日期與會議時間衝突）
- **昨天的 carryover**（明確標示，不要悄悄併入今天的新項目）
- **待釐清**（模糊到無法自信排序的項目）

## Safety Constraints

- 絕對不能捏造輸入內容中沒有的截止日期、寄件人意圖、或急迫程度——沒有明說的急迫性只能是 Hypothesis，不能當成 Fact 呈現。
- 不能悄悄漏掉任何輸入項目——每個項目最後都要出現在優先序清單、carryover、或待釐清區塊之一。
- 不能代替使用者決定要不要出席、拒絕、或改期任何會議——這個 skill 只是提出一個優先序視角，排程動作永遠由使用者自己決定。

## Missing-Data Behavior

- 如果沒有提供行事曆資料，要明確標示 `未提供行事曆資料`，並只用剩下的輸入產出優先序清單，不要憑空猜測會議時間。
- 如果某個項目找不到可判斷急迫性的線索，要標示 `證據不足，無法排序`，放進待釐清區塊，而不是預設為低優先度。

## Self-Review Checklist

- [ ] 每個輸入項目都剛好出現在一個輸出區塊（優先序清單、carryover、或待釐清）
- [ ] 每個排序項目都有說明理由和 Fact/Hypothesis 標記
- [ ] 行事曆衝突有明確標出，而不是只隱含在行程列表裡
- [ ] 沒有任何急迫性或截止日期是在沒有 Source ID 的情況下被當成 Fact 呈現
- [ ] 輸出沒有採取任何排程動作，只有提出建議視角

## Anonymized Eval Case

### Scenario

虛構的 EM「R 經理」提供以下資訊：9:00-11:00 有三場背靠背、沒有緩衝時間的 1-1；一封「某位主管」寄來、沒有明確截止日期、寫著「有空再給我一些意見」的信；一張今天到期的 ticket `ENG-991`；以及昨天沒做完、沒有明確截止日期的 carryover 項目「審查 Falcon 專案的設計文件」。

### Expected Behavior

- `ENG-991` 排序偏高，並附上明確理由（有明說的今天到期日，附 Source ID）。
- 那位主管的信被標示為 Hypothesis 等級的急迫性（「沒有明確截止日期；可能因寄件人資深而隱含急迫性」）——不能悄悄被當成 Fact 拉高成高優先度。
- 三場背靠背 1-1 被明確標示為無緩衝時間的風險。
- 「審查設計文件」的 carryover 項目要明確標示為 carryover，不能跟今天的新項目混在一起，如果無法推斷截止日期就放進待釐清區塊。

### Failure Modes Tested

- [ ] 輸出是否把主管的信在沒有明確截止日期的情況下排成高優先度的 Fact？（不應該）
- [ ] 背靠背會議的風險是否沒有被標出？（不應該）
- [ ] carryover 項目是否被悄悄併入今天的新項目？（不應該）
- [ ] 這個 skill 是否採取或暗示任何排程動作（例如「我已經幫你改時間了」）？（不應該）
